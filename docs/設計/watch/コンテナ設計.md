# Watch コンテナ設計

この文書は、Watch および Chat から利用するローカル依存サービスのコンテナ構成を定義する。

## 方針

DesktopAgent はローカル完結型アプリケーションとする。

ユーザーの画面情報、会話内容、LLM 入出力、音声合成データ、ログは外部ホスティングサービスへ送信しない。必要な処理はすべてユーザーのローカルマシン内で完結させる。

初期設計では、DesktopAgent 専用の API サーバーは置かない。バックエンド側に集約すべき独自処理がまだ少ないため、Unity が制御主体となり、ローカルで起動している各コンテナへ直接接続する。

## 実行構成

Unity はコンテナ化しない。

- 本番実行: Unity のビルド済み exe をホスト上で起動する
- ローカル開発: Unity Editor をホスト上で起動する

Docker で起動するのは、Unity から利用する依存サービスだけとする。

- Ollama
- VOICEVOX
- MySQL

## サービス構成

### Unity

Unity は DesktopAgent のアプリケーション本体であり、Watch と Chat の制御主体である。画面取得、ユーザー操作、キャラクター表示、Chat などの明示的な発話の音声再生、LLM/音声合成/永続化の呼び出しを担当する。

外部通信は、ローカルコンテナへの接続だけとする。

- Ollama: 画面理解、文脈判定、リアクション生成、Chat 応答生成を依頼する
- VOICEVOX: Chat などの明示的な発話テキストの音声合成を依頼する
- MySQL: 設定、ログ、将来の永続データを読み書きする

初期設計では、Unity と各コンテナの間に専用 API サーバーを挟まない。

### Ollama

Ollama は LLM 推論を担当する。Unity から受け取ったプロンプトと画像をもとに、画面理解、文脈判定、リアクション生成、Chat 応答生成を行う。

接続先の初期値は次とする。

Ollama のモデルデータは Docker volume に永続化する。

### VOICEVOX

VOICEVOX は音声合成を担当する。Unity から受け取った Chat などの明示的な発話テキストと話者設定をもとに、音声データを生成する。Watch リアクションはテキスト表示のみとし、VOICEVOX へ送らない。

音声再生、発話キュー、発話の割り込み制御は Unity 側で扱う。

### MySQL

MySQL は将来の永続化に備えて用意する。

初期設計では、Watch 文脈そのものはプロセス内メモリで保持し、MySQL へは永続化しない。MySQL は設定、ログ、将来の履歴保存、ユーザー設定、キャラクター設定の保存先として使う。

MySQL のデータは Docker volume に永続化する。

## dev/prd 構成

コンテナ構成は、開発用 `dev` と運用用 `prd` を分ける。

開発用と運用用では、Compose プロジェクト名、コンテナ名、MySQL 用 volume、ログ、公開ポートを分ける。これにより、開発中の DB、ログ、設定が運用用データと混ざらないようにする。

Ollama のモデルデータはサイズが大きく、dev/prd で分けると同じモデルを二重に保持することになるため共有する。

### dev

dev は Unity Editor から利用する。

dev の Compose プロジェクト名は次とする。

```text
desktop-agent-dev
```

dev の volume は次のように分ける。

```text
desktop-agent-dev-mysql-data
```

### prd

prd は Unity のビルド済み exe から利用する。

dev と同じホスト上で同時起動する可能性を考慮し、prd は dev と異なる公開ポートを使う。

prd の Compose プロジェクト名は次とする。

```text
desktop-agent-prd
```

prd の volume は次のように分ける。

```text
desktop-agent-prd-mysql-data
```

### 分離ルール

dev と prd は次の単位で必ず分離する。

- Compose プロジェクト名
- コンテナ名
- MySQL 用 Docker volume
- MySQL データ
- ログ
- Unity 側の接続設定

Ollama モデルデータは dev/prd で共有する。

Unity Editor は dev の接続先を使い、Unity exe は prd の接続先を使う。

開発中に prd データを直接参照しない。運用中に dev データを直接参照しない。

### 公開ポート

Unity がホスト上から直接接続するため、各サービスはホストへポートを公開する。

dev の公開ポートは次の通りとする。

```text
ollama   10001
voicevox 10002
mysql    10003
```

prd の公開ポートは次の通りとする。

```text
ollama   20001
voicevox 20002
mysql    20003
```

### Volume

Ollama のモデルデータ用 volume は dev/prd で共有する。

```text
desktop-agent-ollama-data
```

MySQL 用 volume は dev/prd それぞれに用意する。

```text
desktop-agent-dev-mysql-data
desktop-agent-prd-mysql-data
```

## セキュリティとプライバシー

DesktopAgent はローカル完結を前提とする。

- Unity はローカルコンテナにだけ接続する
- Ollama、VOICEVOX、MySQL はローカル Docker Compose 上で動作する
- 画面キャプチャ画像は外部へ送信しない
- LLM 入出力は外部へ送信しない
- 音声合成テキストと音声データは外部へ送信しない
- ログは外部へ送信しない

ポートは Unity からの接続のためにホストへ公開するが、外部ネットワークへの公開を目的としない。

## 将来の拡張

初期設計では API サーバーを置かない。

ただし、将来 Unity から直接扱うには重い処理が増えた場合や、永続化、ログ分析、複数プロセス連携を分離したくなった場合は、DesktopAgent 専用 API サーバーの追加を再検討する。

その場合でも、外部ホスティングではなく、ユーザーのローカルマシン上で動くコンテナとして追加する。
