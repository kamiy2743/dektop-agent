# Watch 設計

この文書は、[要件.md](C:\Users\kamiy\WorkSpace\desktop-agent\docs\要件.md) にある Watch 関連要件を実現するための設計をまとめる。

## 目的

Watch は、ユーザーが選択したモニターを継続的に観察し、現在の作業状況を軽く理解し、必要なときだけ短いリアクションを返す常駐処理である。

Watch はリアルタイム同期処理ではない。画面取得、差分判定、画像認識、文脈判定、リアクション生成には必ず遅延があるため、常に少し古い理解になることを前提に設計する。

そのうえで Watch が守るべきことは次の 2 点である。

- 現在の作業と矛盾するほど古い理解を、現在判断に使わないこと
- 重い推論を毎回走らせず、常駐処理として負荷を抑えること

## 設計方針

Watch は次の 3 層に分けて構成する。

- 観察入力層: 画面取得、差分判定、観察起動判定を担当する
- 理解層: 画面理解、文脈判定、記憶更新を担当する
- 反応層: リアクション要否判定と短文生成を担当する

この分離により、画面取得は短周期で続けつつ、重い処理は必要時だけ動かせるようにする。

## Watch の責務

Watch の責務は次の通りとする。

- 対象モニターを一定間隔で観察する
- 画面差分を見て、理解更新が必要かを判定する
- 現在画面の中心的な作業内容を要約する
- 直近の作業文脈と比較し、継続、転換、判定不能を判定する
- 必要なときだけ短いリアクションを返す

一方で Watch は次を責務に含めない。

- ユーザーが明示的に話しかけたときの重い会話処理
- 長期的な個人記憶の保持
- 見えていない情報や意図の推測

## ループ構成

Watch はポーリング型の定期ループで実装する。各周期では次の順に処理する。

1. 対象モニターの最新画面を取得する
2. 前回キャプチャとの差分を測る
3. 差分判定項目と前回認識完了からの経過時間を見て、理解更新を起動するか決める
4. 起動対象なら理解要求を発行する
5. 完了した理解結果があれば、現在の画面と文脈に照らして、文脈更新に使えるかを判定する
6. 文脈更新に使える結果だけを記憶へ反映する
7. 反応条件を満たす場合だけ、理解結果に含まれる Watch リアクションを採用する

### 認識起動

理解更新は毎周期必ず走らせない。差分判定では次の情報を使う。

- キャプチャ画像の差分率
    - 直前のポーリングで取得した前回キャプチャと今回キャプチャを比較して算出する。差分率がしきい値を超えた場合は、画面内容が変化したものとして理解更新を起動する。
- アクティブウィンドウ名、対象モニターの変化
    - 画像差分率に関わらず強い差分として扱う。これは、画面差分が小さく見えても、作業対象や観察対象そのものが切り替わっている可能性が高いためである。
- 前回の認識完了からの経過時間
    - 静止画面でも一定時間ごとに再観察するために使う。大きな差分がなくても、前回の認識完了から 60 秒が経過した場合は理解更新を起動する。
- 判定不能後の再認識間隔
    - 直近の認識結果が判定不能だった場合に使う。1 回目の判定不能後は、通常ポーリングを待たずに即時に最新画面を再キャプチャし、新しい理解要求として認識する。連続する 2 回目以降の判定不能では 10 秒ごとに再認識可能とする。

判定不能の結果が返った時点で最新観察候補がある場合は、即時再キャプチャよりも既存の最新観察候補を優先し、理解要求へ昇格する。最新観察候補がない場合だけ、即時に最新画面を再キャプチャして理解要求を作る。ただし、実行中の理解要求がある場合は追加の要求を積まない。

理解更新は、次のいずれかを満たす場合に起動する。

- キャプチャ画像の差分率がしきい値以上である
- アクティブウィンドウ名が変化した
- 対象モニターが変化した
- 前回の認識完了から 60 秒が経過した
- 判定不能後の再認識条件を満たした

この方式により、静かな画面では負荷を抑えつつ、作業対象の切り替わりや、止まって見える画面の変化も拾えるようにする。

認識処理の時刻と文脈更新の時刻は分けて扱う。

- `lastRecognitionCompletedAt`: 認識処理が完了した時刻
- `lastContextUpdatedAt`: 認識結果を現在文脈に採用した時刻

時間起点の再認識には `lastRecognitionCompletedAt` を使う。`lastContextUpdatedAt` は、文脈が最後に更新された時刻の表示、ログ、調査に使う。

## 観察系列識別子

観察系列識別子は、継続している作業系列または観察系列を表す ID とする。非同期認識要求そのものを識別する ID ではない。

たとえば、VS Code から YouTube に移り、その後 VS Code に戻った場合、最後の VS Code は元の作業への復帰である可能性はあるが、観察系列としては新しい系列として扱う。

```text
VS Code -> YouTube -> VS Code
series-001 -> series-002 -> series-003
```

このように系列を再利用しないことで、作業の継続と転換を区別しやすくする。

### 発行者

観察系列識別子は Watch 側が発行し、プロセス内メモリで管理する。LLM は観察系列識別子を直接発行しない。

LLM は、同一ウィンドウ内で主題が変わったか、または過去文脈へ復帰した可能性があるかを判定する補助として使う。Watch 側は、LLM の文脈判定を根拠にして観察系列識別子を発行する場合がある。

### 発行タイミング

Watch は次の場合に新しい観察系列識別子を発行する。

- Watch 開始時
- 対象モニターが変化したとき
- アクティブウィンドウ名が変化したとき
- 同じ対象モニター、同じアクティブウィンドウ内でも、LLM の文脈判定が `transition` を返したとき

アクティブウィンドウが変化した後に元のウィンドウへ戻った場合も、以前の系列識別子は再利用せず、新しい系列識別子を発行する。

同じ対象モニター、同じアクティブウィンドウ内の理解結果が `transition` の場合は、新しい観察系列識別子を発行し、その理解結果を新しい系列の初期文脈として採用する。

### 理解要求識別子

非同期の理解要求は `recognitionRequestId` で識別する。キャプチャと理解要求は 1 対 1 とし、初期設計では `captureId` は持たない。

`recognitionRequestId` には少なくとも次の情報を対応付ける。

- キャプチャ時刻
- 要求時刻
- 要求時点の観察系列識別子
- 対象モニター ID
- アクティブウィンドウ名
- キャプチャ画像の差分率

判定不能後の即時再認識でも、同じキャプチャは使い回さない。最新画面を再キャプチャし、新しい `recognitionRequestId` を発行する。

最新観察候補は、まだ LLM に投げていない最新キャプチャ候補であり、`recognitionRequestId` を持たない。最新観察候補は、キャプチャ時刻、対象モニター ID、アクティブウィンドウ名、キャプチャ画像の差分率、観察系列識別子などを持つ。

実行中の理解要求が完了した時点で最新観察候補がある場合は、その候補を理解要求へ昇格し、その時点で `recognitionRequestId` を発行する。理解要求へ昇格した後は、候補の観察情報を固定して扱う。

最新観察候補の観察系列識別子が、昇格時点の現在の観察系列識別子と一致しない場合、その候補は理解要求へ昇格せず破棄する。

### 維持する条件

Watch は次のような変化では、同じ観察系列識別子を維持する。

- 軽微なスクロール
- カーソル移動
- 点滅
- 短文追記
- 同じウィンドウ内での小さな表示更新
- 同じ主題内での内容の前進

これらは画面差分が発生していても、途切れず続いている同一系列の観察として扱う。

### 利用方法

理解要求には、発行時点の観察系列識別子を `requestSeriesId` として対応付ける。結果が返った時点で、`requestSeriesId` と現在の観察系列識別子を照合する。

初期設計では、理解要求に紐づく `requestSeriesId` が、結果返却時点の現在の観察系列識別子と一致していることを、現在文脈更新とリアクション生成に使うための必須条件とする。`requestSeriesId` が一致しない理解結果は、現在文脈更新やリアクション生成には使わず、ログに残すだけとする。

過去作業への復帰は、観察系列識別子の再利用ではなく、文脈判定側で扱う。復帰判定が必要になった場合でも、新しい観察系列識別子を持つ現在系列として扱い、過去系列との関連付けは別の文脈情報として表現する。

## 鮮度と結果利用

Watch の理解結果は常に少し古い。このため、設計上は「完全に最新であること」ではなく「現在の作業と矛盾しないこと」を重視する。

### 基本原則

- 結果が少し古いこと自体は許容する
- ただし、別の作業へ転換した後に旧作業の理解を現在判断へ使わない
- リアクションでは、古さが露呈しやすい断定表現を避ける

### 前提

ここで扱う理解結果は、返却時点ではすでに過去画面に対する情報である。

そのため、結果を現在画面そのものとして確定するのではなく、次の 2 用途に使ってよいかを分けて判定する。

- 文脈更新に使ってよいか
- リアクション生成に使ってよいか

文脈更新に使える範囲は比較的広いが、リアクション生成に使える範囲はそれより狭い。

### 結果利用の機械ゲート

理解要求には、その要求を発行した時点の観察情報を対応付けておく。結果が返った時点で、まず次の機械ゲートを確認する。

- `requestSeriesId` が現在の観察系列識別子と一致する
- 対象モニター ID が現在の対象モニター ID と一致する
- アクティブウィンドウ名が現在のアクティブウィンドウ名と一致する

このすべてを満たす場合だけ、機械ゲートを通過したものとして扱う。1 つでも満たさない理解結果は、現在文脈更新やリアクション生成には使わず、ログに残すだけとする。認識要求を出してから結果が返るまでの最大許容時間は設けない。

機械ゲートを通過した理解結果だけを、文脈更新とリアクション生成の候補にする。そのうえで、LLM の画面理解結果と文脈判定を使って、継続、転換、判定不能を扱う。

同じ対象モニター、同じアクティブウィンドウ内の理解結果が機械ゲートを通過し、LLM の `contextDecision` が `transition` の場合は、新しい観察系列識別子を発行する。その理解結果は、新しい観察系列識別子の初期現在文脈として採用する。

ここでいう「文脈更新に使ってよい」とは、その結果を現在画面そのものとして断定するのではなく、現在の作業文脈を更新する材料として使ってよい、という意味である。

機械ゲートは非同期結果の混入を防ぐための判定であり、作業カテゴリ、注目対象、主題、復帰、意味的な矛盾などの判断は LLM の文脈判定で扱う。

### リアクション生成に使うかの判定

文脈更新に使える結果でも、そのままリアクション生成に使えるとは限らない。リアクションはズレが目立ちやすいため、より慎重に扱う。

次の条件を満たす場合だけ、理解結果をリアクション生成にも使ってよい。

- 結果利用の機械ゲートを通過している
- 文脈判定が判定不能ではない
- LLM が `reactionText` に`null`以外を返している

意味的に現在画面と矛盾しないか、発話内容が不自然でないか、直近リアクションと似すぎていないかは LLM に判断させる。LLM はリアクション不要と判断した場合、`reactionText` に`null`を返す。

### 最新優先の扱い

Watch は backlog を溜め続けない。理解処理が重い場合でも、保留中の候補は増やし続けず、常に最新寄りの候補へ寄せる。

- 実行中の理解要求は 1 件まで
- その間に新しい観察候補が出た場合は、最新候補 1 件だけ保持する
- 古い候補を列として積み上げない

これにより、追従性を保ちつつ、古い理解の大量流入を防ぐ。

## 画面理解

画面理解、文脈判定、復帰判定、リアクション生成は、初期設計では 1 回の LLM 呼び出しでまとめて行う。LLM には現在キャプチャ、現在文脈、直近文脈、直近リアクション履歴、機械判定情報を渡す。

LLM は JSON として少なくとも次の情報を返す。

- `workCategory`: 作業カテゴリ。作業の大分類として現在文脈、観察履歴、UI 表示に使う。
- `focusTarget`: 注目対象。画面上の中心対象を、ファイル名、ページ名、動画タイトル、スレッド名など、できるだけ区別可能な名前や特徴として返す。
- `focusSummary`: 注目内容要約。現在その対象について何をしているかを短く説明する。
- `stableHints`: 安定手がかり。継続判定や復帰判定に使える文字列や特徴を配列で返す。
- `visibleTexts`: 画面上の主要テキスト。画面から読み取れる主要テキストを配列で返す。
- `contextDecision`: 文脈判定。`continue`、`transition`、`unknown` のいずれかを返す。
- `contextDecisionReason`: 文脈判定理由。`contextDecision` の理由を短く返す。
- `resumedContextId`: 復帰先の直近文脈 ID。復帰ではない場合は `null` 。
- `reactionText`: Watch リアクションとして画面に表示する短い一言。リアクション不要の場合は`null`。

継続判定に使える安定した手がかりとは、画面差分が多少あっても同じ作業を続けているかを判断しやすい情報である。たとえば次の情報を含める。

- ブラウザの検索バーやアドレスバーに見えているテキスト
- YouTube などの動画ページで見えている動画タイトル
- エディタで見えているファイル名やプロジェクト名
- ドキュメントや資料で見えているタイトル、見出し、ページ名
- チャットやメールで見えているスレッド名、宛先、件名

ここで重要なのは、見えていない情報を補完しないことである。理解結果はあくまで観察可能な範囲に限定する。

LLM が `contextDecision` に `unknown` を返す場合、`reactionText` は必ず`null`とする。

## 文脈判定

理解結果は単独で終わらせず、現在文脈と比較して作業の流れとして解釈する。初期設計では、判定区分を次に絞る。

- 継続: 同じ作業が続いている
- 転換: 主題が別のものに切り替わった
- 判定不能: 十分な根拠がない

詳細化は独立した判定区分としては扱わず、継続に含める。同じ主題の中で内容が前進した場合は、現在文脈を維持したまま注目内容や要約を更新する。

割り込みは、独立した判定区分としては扱わず、転換として扱う。ここでいう転換は、過去文脈の完全な破棄ではなく、現在メイン文脈の切り替えである。転換前の現在文脈は直近文脈へ退避し、復帰判定に使えるようにする。

文脈判定には、Watch 側が機械的に判断する条件と、LLM の `contextDecision` によって判断する条件がある。

機械的に転換する条件は次の通りである。

- 対象モニターが変化した
- アクティブウィンドウ名が変化した

LLM の `contextDecision` で転換する条件は次の通りである。

- 作業カテゴリが変わった
- 注目対象が明確に変わった
- 同じアプリケーション内で、ページ、ドキュメント、スレッド、動画、ファイルなどの中心対象が変わった
- 旧文脈を維持すると現在画面と意味的に矛盾する

### 継続

継続は、現在文脈と同じ作業が続いているとみなせる状態である。次のような場合は継続として扱う。

- 対象モニターとアクティブウィンドウ名が変化していない
- LLM の `contextDecision` が `continue` である
- 作業カテゴリが同じである
- 注目対象が同じ、または強く関連している
- ファイル名、ページ名、スレッド名、見出しなどの安定した手がかりが一致している
- 同じ主題内で入力、スクロール、編集、閲覧位置の変化が起きている

継続では現在文脈を維持し、注目内容、主要テキスト、継続時間などを更新する。リアクションするかどうかは、LLM が返す `reactionText` とリアクション採用条件で判断する。

### 転換

転換は、現在文脈とは別の作業へ切り替わったとみなせる状態である。次のような場合は転換として扱う。

- アクティブウィンドウ名が変化した
- 対象モニターが変化した
- LLM の `contextDecision` が `transition` である

転換では、現在文脈を直近文脈へ退避してから、新しい作業を現在文脈にする。同じウィンドウ内での主題転換によって転換と判断した場合の観察系列識別子の発行順序は、結果利用の機械ゲートに従う。

転換後の作業が直近文脈のいずれかと似ている場合は、過去作業への復帰として扱う。復帰しても観察系列識別子は再利用しない。復帰判定は、以前の文脈と現在の理解結果のアプリ、作業カテゴリ、注目対象、ファイル名、ページ名、スレッド名、主要テキストなどの安定した手がかりを比較して行う。

### 判定不能

判定不能は、現在文脈へ反映するだけの根拠が足りない状態である。次のような場合は判定不能として扱う。

- 読み取れるテキストが少ない
- 画面が暗い、ローディング中、遷移中である
- キャプチャに失敗している
- 注目対象が特定できない
- 差分はあるが、意味づけできない
- LLM の出力が曖昧である

判定不能では現在文脈を更新せず、リアクションも生成しない。1 回目の判定不能後は即時に再認識可能とし、連続する 2 回目以降の判定不能では 10 秒ごとに再認識可能とする。再認識時の候補選択は、認識起動の判定不能後の再認識間隔に従う。

## 記憶保持

Watch の記憶は、長期保存用の記憶ではなく、現在の作業を追うための短中期状態である。

### 保持方式

- Watch の記憶はプロセス内メモリだけに保持する
- Watch 文脈は永続化しない
- Stop 時には破棄する

永続化対象は Watch 記憶ではなく、設定とログに限定する。

### 保持する状態

Watch は少なくとも次の状態をメモリに持つ。

- 観察履歴
- 現在文脈
- 直近文脈
- 反応履歴
- 実行中の理解要求
- 最新観察候補

### 保持量

初期設計では、保持量に上限を設ける。

- 観察履歴: 直近 5 から 10 件
- 反応履歴: 直近 3 から 5 件
- 現在文脈: 1 件
- 直近文脈: 直近 5 件、保持期限 30 分
- 実行中の理解要求: 1 件
- 最新観察候補: 1 件

実際の件数は設定値ではなく実装定数として始め、必要なら後から調整可能にする。

### 現在文脈の中身

現在文脈は、直近の観察結果をそのまま保存するものではなく、今メインで続いているとみなす作業状態である。作業が明確に切り替わった場合、現在文脈は新しい作業へ切り替える。

現在文脈には少なくとも次を含める。

- 観察系列識別子
- 作業カテゴリ
- フォーカス中のアプリケーション名
- 注目対象
- 注目内容要約
- 継続判定に使える安定した手がかり
- 継続時間

### 直近文脈の中身

直近文脈は、転換によって現在文脈ではなくなった作業状態を短期間だけ保持するものである。過去作業への復帰判定に使う。

直近文脈には、現在文脈と同じ要約情報に加えて、直近文脈 ID、観察系列識別子、現在文脈から外れた時刻を含める。直近文脈は最大 5 件まで保持し、現在文脈から外れてから 30 分を超えたものは復帰判定に使わない。

### 観察履歴の中身

観察履歴には生画像ではなく、要約済みの軽量データだけを保持する。少なくとも次を含める。

- 観察時刻
- 観察系列識別子
- 作業カテゴリ
- フォーカス中のアプリケーション名
- 注目対象
- 注目内容要約
- 継続判定に使える安定した手がかり
- 主要テキスト抜粋
- 反応有無

保持しないものは次の通りとする。

- セッションをまたぐ Watch 文脈
- 生キャプチャ画像の履歴保持
- 無制限な全文テキスト履歴

## リアクション

リアクションは、理解できたから必ず返すものではない。初期設計では、リアクション要否の判断と短文生成は、画面理解と同じ LLM 呼び出しで行う。

### 反応する条件

- 結果利用の機械ゲートを通過している
- `contextDecision` が `unknown` ではない
- `reactionText` が`null`ではない

Watch 側では、最小表示間隔や復帰時の強制抑制などの機械的なリアクション抑制ルールは初期設計では持たない。LLM には直近リアクション履歴を渡し、不要な場合は `reactionText` に`null`を返させる。

判定不能、根拠が弱い、直近と似た内容や言い回しになりそう、反応する価値が薄い、現在画面と意味的に矛盾しそう、といった場合は、LLM が `reactionText` を`null`にする。

### 表現上の原則

- システム通知や分析メモのような文体にしない
- キャラクターがそばで見ている軽い一言に留める
- 現在画面と矛盾する断定を避ける
- 同じ言い回しを続けない

## 設定とログの境界

Watch 記憶と混同しないよう、設定とログは別に扱う。

- 設定: 対象モニター、観察間隔、差分しきい値、利用モデル、ミュート状態などの永続化対象
- ログ: 観察実行、推論起動、結果利用判定、反応有無、エラー、処理時間などの永続化対象
- Watch 記憶: 実行中の判断のためだけに持つメモリ状態

ログには生キャプチャ画像を保存しない。通常ログはテキスト、数値、構造化データだけを保存する。

ログには LLM の JSON 出力を保存する。

- `workCategory`
- `focusTarget`
- `focusSummary`
- `stableHints`
- `visibleTexts`
- `contextDecision`
- `contextDecisionReason`
- `isResumedFromRecentContext`
- `resumedContextId`
- `reactionText`

ログには機械判定情報も保存する。

- `seriesId`
- `recognitionRequestId`
- `targetMonitorId`
- `activeWindowName`
- `captureDiffRate`
- `recognitionTriggerReason`
- `mechanicalGateResult`
- `discardReason`
- `requestedAt`
- `capturedAt`
- `lastRecognitionCompletedAt`
- `lastContextUpdatedAt`

ログ確認は、元ログを直接読む運用にはしない。初期設計では `watch-log analyze` のようなログ分析コマンドを専用手順として用意し、期間、観察系列識別子、エラー、破棄理由などで絞り込んで確認できるようにする。アプリ内ログビューは初期設計の対象外とする。
